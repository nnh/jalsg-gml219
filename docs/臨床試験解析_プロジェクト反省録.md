## 1. 何が起きたか（手戻りのまとめ）

JALSG-GML219 では、2025年12月に初版プログラムを作成した後、2026年5月に再実行したところ多数の問題が顕在化し、大幅な修正が必要になった。

|---|---|---|
| 1 | Mac 環境のパスが残っており Windows で動かない | 軽微 |

## 2. 原因分析：なぜ手戻りが発生したか

### 2.1 「とりあえず動く」プログラムを先に作った

初版プログラムは Mac 環境・特定フォルダパス・暫定ロジックで作成されており、「本番で動く状態」ではなかった。再実行時に初めて問題が顕在化した。

**教訓**: 初版でも「実際の実行環境・実際のデータパスで動作確認する」を必須にする。

### 2.2 エンドポイント定義を実装前に PRT/SAP で確認しなかった

- 打ち切り日: データカットオフ日でなく個人の最終追跡日を使うことが常識だが、実装時に確認しなかった
- 治療不応: DS ドメインの中止理由から判定しようとしたが、DS の記録は行政的判断を含むため医学的な「治療不応」と一致しない
- RFS 起算日: PRT に「LFS 到達日」と明記されているのに `crdt`（CR到達日）で実装

**教訓**: エンドポイントを1つずつ実装前に仕様書（下記チェックリスト参照）を作成してレビューする。

### 2.3 データセットの変数名を確認しないまま解析プログラムを書いた

Table1 では `bl_chromdel5qyn` 等の古い変数名を使っており、現データセットには存在しなかった。CSVtoSASDS を確認すれば `del5q` という変数名になっていることがわかった。

**教訓**: 解析プログラム作成前に `proc contents` でデータセットの変数一覧を出力し、それを参照しながらプログラムを書く。

### 2.4 SAP の全出力リストを最初に確認しなかった

Table1 に含めるべき CGA・ELN2017・CCI・血液異常既往・感染既往等を後から追加することになった。SAP 5.2.1 を最初から読んでいれば一度で済んだ。

**教訓**: 解析着手前に SAP の全表・全図リストを作成し、出力仕様を確認してから実装に入る。

### 2.5 ツール固有の罠を知らなかった

PowerShell のダブルクォート here-string `@"..."@` が `$200.` を変数として展開し空文字に置換してしまった。これは SAS 解析固有のトラップで、事前に知っていれば防げた。

**教訓**: 新しいツール（Claude Code + PowerShell + SAS）の組み合わせを使う前に、文字コード・エスケープ等の注意点を共通ナレッジとして整備する。

## 3. 次回プロジェクト開始前チェックリスト

### Phase 0: 文書確認（実装着手前に必須）

- [ ] PRT（試験実施計画書）の主要・副次エンドポイント定義を読む
  - EFS/OS/RFS の起算日・イベント定義・打ち切り定義
  - CR/LFS の定義（5条件 vs 2条件）
  - FTMCC・LACK OF EFFICACY 等の中止理由の取り扱い
- [ ] SAP の全出力表・全図リストを作成する（表番号、内容、集団、変数一覧）
- [ ] aCRF で各 SDTM ドメインの構造（SPID/TESTCD 等の値）を確認する

### Phase 1: データ確認（CSVtoSASDS 実装前に必須）

- [ ] SDTM 各ドメインの CSV を開いて実値を確認する
  - DS: `DSSPID` の値一覧、`EPOCH` の値一覧 → 打ち切り日と中止日を区別
  - RS: `RSSPID` の値一覧、`RSORRES` の値一覧 → CR 判定に使う評価時点を確認
  - AE: `AESTDTC`・`AESER`・`AETOXGR` の存在確認
  - LB/FA/MB: 集計に使うコード（`LBTESTCD`、`FATESTCD` 等）を確認
- [ ] 症例数を確認する（登録例、FAS、PPS）
- [ ] データカット日と最終追跡日の違いを確認する

### Phase 2: 実装設計（コーディング前）

- [ ] エンドポイント実装仕様書を作成する
  ```
  エンドポイント: EFS
  起算日: 登録日（rfstdt）
  イベント: 再発（CE）/ 治療不応（RS: crfl="N"）/ 死亡（DS withdrawal）
  打ち切り: DS withdrawal の DSSTDTC（個人の最終追跡日）
  注意: FTMCC は RS でCR確認 → DS中止理由を参照しない
  ```
- [ ] CSVtoSASDS のセクション設計図を作る（何を §N で実装するか）
- [ ] 全解析プログラムの一覧を SAP と紐付けて作成する

### Phase 3: 実装・確認

- [ ] CSVtoSASDS を実行し `proc contents` で変数名一覧を出力してから解析プログラムを書く
- [ ] エンドポイント変数の整合性チェック SQL を書く
  ```sas
  proc sql;
    select count(*) as n_cr from gml219 where crfl="Y";   /* CRフラグ */
    select count(*) as n_rfs from gml219 where rfs_d ne .; /* RFS期間が存在 */
    /* n_cr = n_rfs になるはず */
  quit;
  ```
- [ ] 特定症例のデータを手で追って結果を確認する（最低 3〜5 例）
- [ ] SAS ログで ERROR 0・WARNING（想定外） 0 を確認してからプッシュ

### Phase 4: 環境設定

- [ ] 開発環境（OS・SAS バージョン・パス）を記録する
- [ ] rawdata パスはバージョン付きサブフォルダを**ハードコードしない**（`rawdata/` 直下に CSV を配置）
- [ ] `.gitignore` に `/input`, `/output`, `/log`, `*.log`, `*.lst`, `*.sas7bdat` を設定する
- [ ] SAS ファイルの文字コードは Shift-JIS（CP932）に統一する

## 4. 理想的な作業フロー

```
1. 文書確認（PRT・SAP・aCRF）
   ↓ エンドポイント仕様書・出力リスト作成
2. データ確認（SDTM CSV の proc freq で実値確認）
   ↓ DS/RS/AE ドメイン構造をメモ
3. CSVtoSASDS 設計・実装
   ↓ proc contents で変数名確認
4. エンドポイント変数の整合性チェック
   ↓ 数例の手追跡
5. 解析プログラム一式を SAP 照合しながら作成
   ↓ 各プログラムを SAS 実行し ERROR=0 を確認
6. GitHub にプッシュ
   ↓
7. 出力 RTF を SAP と照合してレビュー
```

フロー全体で最も手戻りが少ないのは **ステップ1〜2に時間をかける** こと。エンドポイント定義の誤りは下流の全プログラムに波及するため、最初に正確に理解することが最優先。

## 5. 今回発生しなかったが次回注意すべきこと

### 5.1 複数環境での実行確認

今回は Mac → Windows の移行問題が発生した。次回から:
- 開発機と実行機のOS・SASバージョンを最初に揃える
- 揃えられない場合は、環境差異（パス区切り文字・文字コード・SASバージョン差異）をリスト化しておく

### 5.2 データ更新時の対応

データカットが更新された場合:
- rawdata の CSV を入れ替えるだけで CSVtoSASDS が再実行できる構造にしておく
- バージョン付きサブフォルダ参照は禁止。`rawdata/XX.csv` を直接参照する

### 5.3 Claude Code + SAS 作業固有の注意

- SAS ファイルの書き出しは **PowerShell + CP932** のみ使用（Edit ツール禁止）
- PowerShell here-string は **`@'...'@`（シングルクォート）** を使う
- `$200.` のような SAS フォーマット指定子を含む場合は書き出し後に必ず検証する
- `working_dir` マクロの `%scan` 区切り文字は `'\'`（シングルクォート）を使う
- `proc report` の `column` 文には全列を明示する（`define` 文があっても `column` に書かないと非表示）
